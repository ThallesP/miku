# miku

A Bun monorepo: a NestJS control-plane API, a TanStack Start dashboard, and Restate workers that join a mesh network and show up as servers on a realtime canvas.

## Layout

| Package | Purpose |
| --- | --- |
| `apps/api` | Control-plane API. [NestJS](https://nestjs.com) + [typia/nestia](https://nestia.io) with a clean architecture (`core` / `domain` / `infra`, modeled after [rocketseat-education/05-nest-clean](https://github.com/rocketseat-education/05-nest-clean)). Serves REST on `:3100` plus SSE at `/events`. |
| `apps/web` | Dashboard. TanStack Start + React Flow canvas, talks to the API through `@miku/sdk` and listens to `/events` for realtime updates. |
| `apps/workers` | Worker process. Joins the network (Tailscale, with a local fallback), registers with the control plane, heartbeats, and serves [Restate](https://restate.dev) handlers on `:9080`. |
| `packages/db` | The model package: MikroORM (libsql/SQLite) entity classes `Application` and `Server` — plain fields plus computed getters (`isOnline`), a validating `create()` factory returning `Result`, the `Position` value object, and `dbConfig`/`initOrm`. Compiled package consumed by the API (and by anything else that needs the model directly, e.g. workers). |
| `packages/sdk` | Type-safe API client generated by `nestia sdk` from the controllers. Used by the dashboard and the workers. |

## Architecture (apps/api)

```
src/
  core/        use-case errors
  domain/      business logic
    canvas/
      application/use-cases    create-application, move-application, register-server, ...
      application/repositories abstract repositories (DI tokens)
      application/events       ChangePublisher port
  infra/       NestJS adapters
    env/       typia-validated env (PORT, MIKU_DB_PATH)
    database/  MikroOrmModule wiring + repository implementations
    events/    rxjs ChangeStream (implements ChangePublisher, feeds SSE)
    http/      one controller per use-case + presenters
```

The entities themselves live in `@miku/db`: a single decorated class is both the domain entity (behavior, factories) and the table definition — no separate persistence records or mappers. Repositories lean on MikroORM's unit of work: each HTTP request gets its own identity map via `RequestContext` (wired by `@mikro-orm/nestjs`), so `save()` is just a `flush()` of whatever the use-case mutated.

Request validation and response serialization are compile-time generated by typia via `@nestia/core` (`TypedRoute`, `TypedBody`, `TypedParam`).

## Start

```bash
bun install
bun run dev
```

This builds the shared packages, then starts the API on `http://localhost:3100`, the dashboard on `http://localhost:3000`, and a worker. The worker registers itself with the control plane and appears on the canvas in realtime.

## SDK

`packages/sdk` is generated — don't edit it. After changing controllers:

```bash
cd apps/api
bunx nestia sdk
```

## Networking (Tailscale)

Without configuration, workers report their LAN address (`network: "local"`). To actually join a tailnet, run `tailscaled` on the worker host and set:

```bash
TS_AUTHKEY=tskey-auth-... WORKER_NAME=worker-1 bun --cwd apps/workers start
```

Other worker env vars: `CONTROL_PLANE_URL` (default `http://localhost:3100`), `RESTATE_PORT` (default `9080`).

## Restate

Workers expose a `worker` service. To orchestrate them, run a Restate server and register the deployment:

```bash
restate-server &
restate deployments register http://localhost:9080
```

## Database

SQLite file at `miku.db` (override with `MIKU_DB_PATH`). Schema is synced automatically on boot — no migrations in dev.
